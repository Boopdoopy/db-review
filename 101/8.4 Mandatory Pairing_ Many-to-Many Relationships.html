    8.4 Mandatory Pairing: Many-to-Many Relationships

    Re-submit Assignment
    • Due Nov 28, 2016 by 10am
    • Points 30
    • Submitting a website url

    Some relationships don't quite fit the one-to-many model. Let's say we work for a company that allows users to rate local businesses online. We'll call it ... Welp, as in "Welp, I would probably eat at that restaurant again, I guess. Three and a half stars."

    In particular, let's examine the data we have for Grundy Hollow, a small and quaint little town that we would strongly recommend visiting if we hadn't made it up just now.

    The chapel of Grundy Hollow, currently holding steady at a whopping 4.8 stars. "Like something out of a postcard!" Etc.

    We also have some Welp users from Grundy Hollow:

    Release 0: Build Your Tables

    do the thing
    1. Create a folder for this project in phase-0-tracks/databases called welp.
    2. In the welp directory, create a database called welp.db.
    3. Create the businesses table. Make sure to use a primary key.
    4. Create the users table. Make sure to use a primary key.
    5. Populate each table with a few items.

    Release 1: Connect the Tables

    We'd like to connect our two tables of data. But a one-to-many relationship won't work, because:

    • A Welp user can review many businesses.
    • A business can have many Welp users as customers.

    Each item in a table could have many of the items in the other table. This is called a many-to-many relationship.

    To set up a many-to-many relationship, we'll need to create a third table. This is often called a junction table or join table. Here's the schema for that relationship:

    This simple table connects users to businesses. It doesn't really have any of its own data -- it just exists to store pairs of user and business ids. It has two foreign keys; optionally, it could also have a primary key. Notice the name of the table: businesses_users. It's a convention to name simple junction tables this way, with the two table names arranged in alphabetical order and separated by an underscore.

    Sometimes, the joining table in a many-to-many relationship has its own data, too. If we chose to connect users and businesses through a reviews table, for instance, each review might have a date, a number of stars, a comment (such as "Their spaghetti is great!") and so on:

    In this case, it makes sense to use a primary key in the reviews table, since each review is a meaningful item in its own right, not just a connection between two other tables.

    do the thing
    1. Build the reviews table depicted in the second schema. When you create the table, you'll need to declare two foreign keys and a primary key, along with any data needed for the review.
    2. Populate your reviews table with a few pieces of data. You'll need to provide valid integers for both of the foreign keys (one valid id of a user, and one valid id of a business). An example review might consist of 5 stars, the text "great food!", a business ID of 1, and a user ID of 1 (provided you have a business with a primary key of 1 and a user with a primary key of 1).

    Release 2: View a Many-to-Many Relationship

    find the thing
    1. In the welp directory, create a file called queries.txt. Once you've figured out the queries below, paste them into queries.txt, along with the output from your database.
    2. You've joined two tables before. How do you join three?
    3. There are some repetitive columns if we choose to view all fields. Choose instead to just view the user's names, the business name, the star rating, and the comment.
    4. How do you view the data of all three tables for just one particular business? 

    Release 3: Draw a Schema

    do the thing

    Come up with an example of a real-world many-to-many relationship. Using whatever medium you prefer (online diagramming tool, pen and paper, etc.), draw a schema showing how you would set up this many-to-many relationship in a database. Save a screenshot or picture of your schema in the welp directory.

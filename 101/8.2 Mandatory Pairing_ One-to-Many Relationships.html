    8.2 Mandatory Pairing: One-to-Many Relationships

    Re-submit Assignment
    • Due Nov 28, 2016 by 10am
    • Points 20
    • Submitting a website url

    Real-life data is usually too complex to fit into a single database table. For instance, software designed to organize a sports league might have the following tables:

    • coaches
    • players
    • games
    • venues
    • teams

    ... and so on. But who is on each team, and which coach is leading them? Which venue is hosting the next game? We need ways to connect rows together in meaningful ways. We need ways to express a relationship between, say, Row 11 of the players table and Row 3 of the teams table.

    Release 0: Build a Table with a Primary Key

    If we're going to refer to particular rows in order to relate them to other particular rows, we need a unique way to identify each row. This unique identifier for a row is called its primary key.

    Consider the roster of roller derby players below. We are delighted to tell you that these names are from real rosters. (If you're unfamiliar with roller derby, think of it as a violent race amongst a horde of roller skaters and you'll be close enough.)

    How can we identify a particular player? You might be tempted to use the player's name or number as a primary key, but remember that most databases have thousands, if not millions, of rows. So let's pretend this is going to be a giant national database of players. In that case, numbers will frequently be repeated, and names might as well. If a player in Indiana and a player in California both decide to use the name "Nemesistah" (and why wouldn't they), we wouldn't be able to tell them apart. Primary keys must be guaranteed to be unique.

    "That database table over there better be using a primary key. Do I have to do everything myself around here?"

    There is one plainly unique option in the roster spreadsheet: the row number. No matter how many players we wind up with, only one player will ever be in row 2 at (or row 2 million!) a time. Many databases use a similar, but not identical, system: they use a counter, incrementing it by one each time.

    An incrementing primary key often visually resembles a row number, but note that primary keys generally do not change. Once a player is created with a primary key of 4, that player will be identified as 4 permanently, even if players 1, 2, and 3 are deleted.

    You can create tables with primary keys in SQLite using the following syntax:

    CREATE TABLE your_table_name (
  id INTEGER PRIMARY KEY,
  ... the rest of your attributes go here ...
);

    In the above example, we've added a primary key called id. It's nice to set up that column first, so it will appear along the left-hand side in the result of any queries we make. Technically we could have called it anything we wanted, but id is a very common column name for a primary key.

    do the thing
    1. Fun fact: If you run sqlite3 derby.db in the phase-0-tracks/databases folder and the derby.db database does not exist, SQLite will create it for you. (This is convenient, but also sometimes wildly confusing. If you ever try to open an existing database only to find it empty, double-check the name before you panic -- you may have gotten the name wrong and created a new database by accident.) Go ahead and run that command to create a roller derby database.
    2. Create a players table that has a primary key, along with fields (columns) for all the other data in our screenshot.
    3. Use your new SQL skills to insert two players into the table. You do not have to insert a value for the primary key -- that's automatic. Just pretend the primary key column doesn't exist at all, and create rows as you typically would.
    4. Select all the players in order to see your work. If you set up your primary key properly, you should see something that looks like a row number (but isn't!) next to your data:

    5. Select just the player with an id of 1.

    Release 1: Create a One-to-Many Relationship

    A database administrator would immediately recognize a problem with our roster data: it isn't normalized, which is database-speak for "it's repetitive."

    Notice how often the strings "Wrecking Belles" and "Team Unicorn" appear in the table. If we had millions of rows like these, all those repetitive strings would take up a decent amount of disk space. Let's say that instead we decided to represent the Wrecking Belles team with a 1 and Team Unicorn with a 2. Using these integers instead of strings would take up less room.

    If we had a teams table, we could do just that: in the players table, we could refer to a given team by its primary key instead of by its name. This sort of database design is often shown using a certain style of illustration called a schema (skee-muh):

     

    Here we've provided a schema for a players table and a teams table, and drawn a line between two of their fields to communicate that a foreign key in the players table is referencing the primary key of the teams table. The vertical arrangement of column names in each table is a little disorienting at first, but you'll want to get used to looking at database schemas -- they're a great tool when designing interconnected systems of tables.

    More generically, a database's schema refers to its configuration of tables and column names. You can run the .schema command in the SQLite console to view a database's table configurations.

    The players table:

    The teams table:

    Note that when a primary key is used in a different table, it's no longer referred to in that context as a primary key. It's a foreign key -- meaning, "This is the primary key of some other table, not this table." Using that terminology, we would say that in the players table, id is a primary key and team is a foreign key. Because we helpfully named the foreign key team_id, it's not hard to figure out that it refers to the id row in the teams table. Such naming conventions are very important for clarity!

    Because one team can have many players, but each player is only on one team (at least in our example), this type of relationship is called a one-to-many relationship. In one-to-many relationships, the foreign key goes in the table that represents the "many" side of the relationship:

    • If one stylist has many clients, the clients table would have the foreign key.
    • If one bus driver has many passengers, the passengers table would have the foreign key.
    • If many constituents have one senator, the constituents table would have the foreign key.
    do the thing
    1. In your roller derby database, create a teams table that uses an integer primary key and has a name column.
    2. Add the two teams to the table, and select the teams so you can verify your work.
    3. Our players table is going to need an update -- the old string data isn't what we want. More sophisticated databases have fancy ways to alter tables without losing data, but unfortunately, SQLite is not a fancy database. Drop your players table and create a new one that has a foreign key column in it instead of a VARCHAR column for team. Here's the syntax for declaring a column and then identifying it as a foreign key, where you would fill in real values for the placeholders shown in < and > symbols: 
      <name_of_column_to_create> <DATA_TYPE_OF_COLUMN_TO_CREATE>,
FOREIGN KEY(<name_of_column_to_create>) REFERENCES <existing_table_name>(<existing_column_name>)

      So to create a foreign key column of team_id that refers to the id column of the teams table, you would define the column this way in your CREATE TABLE query:

      CREATE TABLE players (
  ... all your other column declarations here, including primary key ...
  team_id INT,
  FOREIGN KEY (team_id) REFERENCES teams(id)
);
    4. Add three players to this table, making sure you provide a valid integer value for the team_id column that matches an existing id in the teams table. Don't put all of the players on the same team.

    Release 2: View a One-to-Many Relationship with Joins

    Though we set up two related tables in the previous releases, we only know how to view one table at a time. Viewing just the players table with SELECT * FROM players isn't particularly helpful in terms of seeing who is on what team, because humans are not computers and it's too much work to remember which integer foreign key refers to which team.

    "I'm on team 1. That's ... the Wrecking Belles, right? Or wait, is it Team Unicorn? I better figure this out before I score points for the wrong team. THINK, Magic Lee Malicious, THINK!"

    In SQL, a special query called a join lets you see data from more than one table.

    try the thing
    1. In your roller derby database, view the players table with SELECT * FROM players. Notice the lack of team names.
    2. Use a join to view the data in both tables at once: 
      SELECT * FROM players, teams WHERE players.team_id = teams.id;
    3. Notice the data in the players table appears on the left. By examining the syntax of the above command, can you figure out how to swap the join so the teams data appears on the left instead?
    4. These queries are showing some repetitive fields, since the foreign key of the players table is the primary key of the teams table. Let's just view the relevant data instead. Notice how we have to specify both the table and the column for each piece of data we want to view, since multiple tables are being queried: 
      SELECT players.name, teams.name FROM players JOIN teams ON players.team_id = teams.id;

    There's a lot more to learn about SQL, but for now, hand in the GitHub URL of your roller derby database file to complete this challenge.

    8.1 Intro to Databases

    Re-submit Assignment
    • Due Nov 28, 2016 by 10am
    • Points 30
    • Submitting a website url

    Relational databases like SQLite store their data in tabular format -- in other words, in tables that use rows and columns to organize data. If you've ever worked with a spreadsheet, you can conceive of a database table as a page in a spreadsheet:

    Notice how each horizontal row represents a rabbit, and each vertical column represents an attribute or property of a rabbit. Database tables store information in the same sort of format.

    You can create, access, update, and delete the data in such a table by

    1. Contacting the database
    2. Submitting a query to the database -- a request to create, access, update, or delete certain information

    There are a few ways to communicate with a database. For these releases, we'll be using SQLite's command-line utility, also sometimes referred to as the console. Most databases have some sort of program like this that allows you to issue queries. The console commands themselves vary slightly from database to database -- for instance, to exit SQLite's console, you type .exit, but to exit some other type of database (Postgres or MySQL are common ones), you might type quit or q.

    As for querying the database, we'll be using SQL, or Structured Query Language. Like console commands, SQL commands can vary a little from database to database. (Some databases don't use SQL at all, but that's a whole other topic.) Most of the SQL commands we give you should work on just about any SQL-supporting database, though.

    Make sure to protect yourself with Git workflow for these releases, in case you make a mistake. That way you won't lose your work if you accidentally delete data. As a professional developer you wouldn't check your database into version control, but SQLite uses small files and it's smart to keep them in version control while you're learning. As long as you're committing, you can always roll back when you make a mistake.

    Release 0: Explore a Table

    The hardworking employees at Fluffy Meadows Animal Sanctuary are exhausted from tracking their residents by hand -- especially the rabbit population, which can get out of control quickly. "I barely have time to pet the baby goats anymore," one employee reported tearfully. In other words, we're gonna need a database.

    do the thing

    1. We've provided a database file in phase-0-tracks/databases. The file is called sanctuary.db. Start the console by running sqlite3 sanctuary.db in your terminal from that directory. You should get a prompt that looks something like this:

      Typing .help will list all the available commands, and .exit will take you back to your regular terminal.

    2. Let's view the available tables in this database with the following console command. (Notice that console commands start with a period and do not end with a semicolon. You'll be alternating between console commands and SQL commands, so it can get a little confusing if you aren't mindfully aware of the differences.)

      .tables

      You should see a rabbits table in the list.

    3. Run this SQL query to see the data in the rabbits table:

      SELECT * FROM rabbits;

      (Notice that SQL commands do not start with a period, and they do end with a semicolon.)

      This data is similar to the data in our spreadsheet screenshot above: we have the name of each rabbit, the age of each rabbit, and a boolean for whether the rabbit is fluffy or not.

    4. Hmm, what's the asterisk in that previous command? It's shorthand for "all fields" or "all columns". Compare the results you got above with the results you get from a more specific command:

      SELECT name, age FROM rabbits;

      Notice that the fluffy column is no longer present because we specifically asked only for the name and age properties.

    5. We can be more specific. Let's just select rabbits that are fluffy:

      SELECT * FROM rabbits WHERE fluffy="true";

    6. Based on the syntax above, select only the rabbit whose name is Queen Bey.

      Queen Bey surveys her admirers from atop her festive fall pedestal.

    7. You may have noticed, in comparing the spreadsheet screenshot with the database table, that Gumdrop is missing from the database. It's not a rabbit party without Gumdrop! Add him to the table:

      INSERT INTO rabbits (name, age, fluffy) VALUES ("Gumdrop", 2, "true");

      Run the appropriate SQL command to view the table data and confirm that Gumdrop has been added, and then add one more rabbit of your choice.

    8. It's Queen Bey's birthday. Use the following SQL command to update her age:

      UPDATE rabbits SET age=4 WHERE name="Queen Bey";

      View the data in the rabbits table to check your work, and then update another rabbit in some way.

    Release 1: Create a Table

    You may have noticed that our rabbit data involved a few data types: strings (known as VARCHAR in database-speak), integers, and booleans. SQLite offers more data types than that, but we'll work with those familiar ones for the time being, and leave other data types as a stretch learning goal.

    Just as you can create rows in a database, you can also use SQL to create entire tables. For instance, here's the SQL we used to create the rabbits table:

    CREATE TABLE rabbits (
  name VARCHAR(255),
  age INT,
  fluffy BOOLEAN
);

    The VARCHAR data type requires a length declaration. For now, 255 characters is a reasonable standard length to use, though you're welcome to explore the nuances of differing VARCHAR lengths on your own. Notice that the command follows the following template:

    CREATE TABLE table_name (
  attr_name DATATYPE,
  attr_name DATATYPE,
  attr_name DATATYPE
);
    do the thing
    • Think of an animal to add to the sanctuary, and create a table for it that has at least four columns and all three data types you've learned.
    • Create a table for these animals in sanctuary.db.
    • Add three or four rows to the table.
    • Update one of the rows with at least one new piece of data.

    Release 2: Research on Your Own

    find the thing

    1. If you haven't already (bonus points if you were ahead of the curve!), find a SQL cheat sheet and a few SQL resources to use for reference and to help you practice.

    2. How do you drop a table? Use your newfound knowledge to drop the hamsters table, as the hamsters have been moved to their own (colorful, plastic) facility.

    3. How do you delete a row based on some condition? Use this command to remove Sylvie from the rabbits table. (Nothing bad happened to her, we promise -- she was just adopted!)

      Sylvie isn't going to apologize for living her best life.
